# Database Migration Guide: Local SQLite to Supabase (Postgres)

This guide outlines the process of migrating the existing local staff database (`portal.db`, SQLite) into the new managed Supabase (Postgres) database.

## 1. Extract Schema and Data from SQLite

The current database is a local SQLite file (`portal.db`). We need to extract both the schema and the data to import them into the new PostgreSQL database.

### Option A: Using `sqlite3` CLI
1. Dump the schema and data:
   ```bash
   sqlite3 portal.db .dump > export.sql
   ```
2. Convert SQLite syntax to Postgres syntax. You will need to manually change:
   - `AUTOINCREMENT` to `SERIAL` (or `GENERATED BY DEFAULT AS IDENTITY` in Postgres).
   - `DATETIME` to `TIMESTAMP WITH TIME ZONE`.
   - Remove SQLite-specific pragmas.

### Option B: Using a Migration Tool (e.g., pgloader)
`pgloader` can automatically translate the SQLite schema to Postgres and migrate the data in one step:
```bash
pgloader portal.db postgresql://postgres:<password>@<db.project.supabase.co>:5432/postgres
```

## 2. Target Supabase Schema

Once the baseline data is imported, ensure the following core tables are established in Supabase. You may need to run `ALTER TABLE` to align your imported SQLite data with the new schema:

```sql
-- Example Schema definition
CREATE TABLE IF NOT EXISTS staff (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  full_name TEXT NOT NULL,
  role TEXT NOT NULL,
  department TEXT,
  contact_info TEXT,
  employment_status TEXT DEFAULT 'active',
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS documents (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  gcs_object_path TEXT NOT NULL,
  display_name TEXT NOT NULL,
  category TEXT,
  owner_staff_id UUID REFERENCES staff(id),
  current_version INTEGER DEFAULT 1,
  last_modified_at TIMESTAMPTZ DEFAULT NOW(),
  checksum TEXT
);

CREATE TABLE IF NOT EXISTS audit_log (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  actor_id UUID REFERENCES staff(id),
  action TEXT NOT NULL,
  target_type TEXT NOT NULL,
  target_id UUID,
  timestamp TIMESTAMPTZ DEFAULT NOW(),
  metadata JSONB
);

CREATE TABLE IF NOT EXISTS users (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  staff_id UUID REFERENCES staff(id),
  email TEXT UNIQUE NOT NULL,
  role TEXT DEFAULT 'viewer',
  password_hash TEXT,
  last_login_at TIMESTAMPTZ
);
```

## 3. Post-Migration Verification

1.  Log into the [Supabase Dashboard](https://app.supabase.com).
2.  Navigate to the Table Editor.
3.  Verify that all staff records from `portal.db` are present.
4.  Ensure that any sensitive data (if present) is properly secured with Row Level Security (RLS) policies if you intend to use direct Supabase client access in the future. (Note: Our current architecture uses a Node.js Backend API running with a Service Role key, meaning the API handles authorization).
